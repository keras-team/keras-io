# Contributing to keras.io

Thank you for your interest in contributing to [keras.io](https://keras.io)!
This document covers the basics you need to get started. It links out to
more detailed guides where they already exist, so please follow those links
rather than skipping them.

## Ways to contribute

- **Report a bug or doc issue** — open an issue using the *Bug report* or
  *Doc improvement* template.
- **Propose or submit a new code example** — check
  [`call_for_contributions.md`](call_for_contributions.md) for examples
  we're actively looking for, then see
  [Adding a new code example](#adding-a-new-code-example) below.
- **Fix something in an existing example** — see
  [Fixing an existing code example](#fixing-an-existing-code-example) below.
- **Improve prose docs** (guides, quickstarts, templates) — regular PRs
  editing the relevant `.md`/`.py` file are welcome.

## Code style

All code examples must follow the
[Keras.io code examples contributor guide](contributor_guide.md). Read it
before opening a PR that adds or modifies an example — it covers variable
naming, imports, model-building conventions, and text/markdown style.

## Running the docs locally

```
pip install -r requirements.txt
pip install -U keras
cd scripts
python autogen.py make
python autogen.py serve
```

Then open `0.0.0.0:8000` in your browser.

Alternatively, with Docker:

```
docker build -t keras-io . && docker run --rm -p 8000:8000 keras-io
```

or via the Makefile:

```
make container-test
```

See the [README](README.md#generating-a-local-copy-of-the-website) for more
detail on these commands.

## Adding a new code example

Code examples on keras.io are **tutobooks**: a single Python script that is
the source of truth, rendered into a notebook, markdown, and webpage.

1. Write your script and format it with `black`.
2. Add the tutobook header (`Title`, `Author`, `Date created`, `Last
   modified`, `Description`, `Accelerator`).
3. Place it in the relevant subfolder, e.g. `examples/vision/script_name.py`.
4. Preview and test it locally:
   ```
   cd scripts
   python autogen.py add_example vision/script_name
   ```
   This is also the test for your example — the command fails if any cell
   takes too long to run, so keep examples lightweight (aim for well under
   20 minutes on a V100 GPU).
5. Submit a PR containing **only** the `.py` file. Do not include the
   generated `ipynb`, `md`, or `img` files at this stage.
6. Once the PR is reviewed and approved, add the generated files
   (`examples/.../ipynb`, `examples/.../md`, `examples/.../img`) to the same
   PR. A maintainer will then merge it.

Full details, including converting from an existing `.ipynb`, are in the
[README](README.md#adding-a-new-code-example).

## Fixing an existing code example

- **Simple typo fixes**: submit one PR that updates the `.py`, `.md`, and
  `.ipynb` files together.
- **More substantial fixes**: submit a PR with only the `.py` file changed.
  The `.md` and `.ipynb` files will be regenerated once the fix is reviewed
  and approved.

## Files you should never hand-edit

The following are autogenerated and must not be modified directly:

- `site/*`, `sources/*`
- `templates/examples/*`, `templates/guides/*`
- `examples/*/md/*`, `examples/*/ipynb/*`, `examples/*/img/*`
- `guides/md/*`, `guides/ipynb/*`, `guides/img/*`

The files you *can* edit by hand are `templates/*.md` (excluding the
autogenerated subfolders above), `examples/*/*.py`, `guides/*.py`,
`theme/*`, and `scripts/*.py`.

## Opening a pull request

Please use the PR template checklist — it's pre-filled when you open a PR
and covers the most common review comments (which files to include, whether
you've previewed your example locally, etc.).

## Getting help

For general Keras usage questions and support (not a bug or doc issue),
please use the [TF Forum](https://discuss.tensorflow.org/tag/keras) instead
of opening an issue here.
